Informix-Plug-in 'Replication' für OAT

IBM® Informix®-Plug-in 'Replication' für OpenAdmin Tool (OAT) bietet eine grafische Oberfläche zum Konfigurieren, Überwachen und Verwalten von Enterprise Replication (ER). Sie können auch Raster verbundener Replikationsserver verwalten.

Voraussetzungen:
Für das Plug-in 'Replication' ist IBM Informix ab Version 11.50.xC4 erforderlich. OpenAdmin Tool (OAT) for Informix erfordert für Connection Manager IBM Informix 11.70.xC3 und den zum Lieferumfang von Informix 11.70.xC3 oder IBM Informix Client Software Development Kit (Client SDK) 3.70.xC3 gehörenden Connection Manager.

Mit dem Plug-in 'Replication' können Sie Enterprise Replication-Server und -Domänen überwachen:

  • Zeigen Sie eine Topologie der ER-Domäne und aller bekannten Knoten an.
  • Zeigen Sie die Zusammenfassung und detaillierte Informationen zu Knoten in der Domäne an.
  • Zeigen Sie detaillierte Informationen zu Replikaten und Replikatsets in der Domäne an.
  • Legen Sie Alertschwellenwerte für die Domäne fest.

Sie können ER-Verwaltungstasks ausführen:

  • Definieren und modifizieren Sie ER-Server, -Replikate und -Replikatsets.
  • Verwenden Sie Vorlagen zum Konfigurieren von ER.
  • Verwalten Sie den Status von Servern, Replikatsets und Replikaten.
  • Resynchronisieren Sie Daten zwischen Replikationsservern.
  • Ändern Sie die ER-Konfigurationsparameter für einen Server.

Sie können Raster verbundener Replikationsserver in einer Domäne erstellen und verwalten:

  • Zeigen Sie Informationen zu den Rastern in einer Domäne an.
  • Erstellen Sie ein Raster und ermöglichen Sie Quellenservern und Benutzern die Ausführung von Rasterbefehlen.
  • Modifizieren Sie ein Raster, um Server hinzuzufügen, aktivieren oder inaktivieren Sie Quellenserver oder berechtigen Sie zusätzliche Benutzer.
  • Überprüfen Sie den Status von Rasterbefehlen und führen Sie fehlgeschlagene Rasterbefehle erneut aus.
  • Leiten Sie Clientverbindungen auf der Basis der Qualität der replizierten Daten an Server im Raster weiter, indem Sie Service-Level-Agreements (SLAs) für eine Connection Manager-Instanz konfigurieren.

Sie können eine Connection Manager-Instanz definieren, um Clientverbindungsanforderungen für eine Kombination von Clustern, Rastern, Replikatsets und Server-Sets zu verwalten. Sie können einer Connection Manager-Instanz Verbindungseinheiten hinzufügen, eine Connection Manager-Instanz starten und stoppen und Service-Level-Agreements (SLAs) erstellen.

Konfigurieren des Plug-ins 'Replication'

Konfigurieren Sie das IBM Informix-Plug-in 'Replication' für OpenAdmin Tool (OAT) für die Überwachung und Verwaltung von Enterprise Replication (ER).

Gehen Sie wie folgt vor, um das Plug-in 'Replication' zu konfigurieren:

  1. Installieren Sie das Plug-in 'Replication'.
  2. Erstellen Sie OAT-Gruppen.
  3. Melden Sie sich bei OAT mithilfe eines der Server in einer vorhandenen Enterprise Replication-Domäne oder mit einem beliebigen Server an, den Sie für Enterprise Replication definieren wollen.

    Weitere Informationen zum Installieren des Plug-ins 'Replication' und dem Erstellen von OAT-Gruppen finden Sie in der Hilfe zu OAT. Klicken Sie im OAT-Menü auf Hilfe > OAT-Taskhilfe, um auf den Hilfetext zuzugreifen.

Überwachen von Enterprise Replication

Mit dem IBM Informix-Plug-in 'Replication' für OpenAdmin Tool (OAT) können Sie Enterprise Replication-Server und -Domänen überwachen.

Anzeigen der Topologie einer ER-Domäne

Zeigen Sie die Topologie oder die Routing-Zuordnung einer Enterprise Replication-Domäne (ER-Domäne) an.

Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > ER-Domäne, um die Topologie der aktuellen Domäne anzuzeigen.

Die Seite Domäne zeigt alle Enterprise Replication-Domänenknoten an, die der aktuelle Knoten erkennt. Beispiel: Ein Endknoten erkennt sich selbst und seinen übergeordneten Knoten. Knoten werden durch den Gruppennamen bezeichnet, der in der Datei SQLHOSTS definiert ist.

Wichtig:
Wenn Sie mit einem Nicht-Root-Knoten oder einem Endknoten verbunden sind, sind die Informationen, die Sie anzeigen können, und die Aktionen, die Sie vornehmen können, beschränkt. Wechseln Sie zu einem Rootknoten in der Domäne, um uneingeschränkten Zugriff zu erhalten.

Auf der Seite Domäne können Sie die folgenden Aktionen ausführen:

  • Ermitteln, ob ein Knoten eine Aktion erfordert, indem visuelle Alerts überprüft werden
  • Namen des Knotens und aller Knotenelemente anzeigen, indem Sie die Maus über das Knotensymbol bewegen.
  • Unterverzeichnisstruktur eines beliebigen Knotens, der kein Endknoten ist, durch Klicken auf das Symbol zum Erweitern anzeigen oder ausblenden.
  • Ihre Anzeige anpassen:
    • Verwenden Sie die Schiebeleiste der Zoomfunktion, um die Domäne zu vergrößern oder zu verkleinern.
    • Klicken Sie zum Anzeigen der Aktualisierungsrate auf das Uhrsymbol.
    • Klicken Sie zum Ändern der Aktualisierungsrate auf das Uhrsymbol und verwenden Sie die Schiebeleiste, um eine Aktualisierungsrate zwischen 0 und 300 Sekunden anzugeben.
    • Klicken Sie zum Anzeigen der Verbindungslinien zwischen den Knoten auf das Symbol Normalsicht.
    • Klicken Sie zum Ausblenden der Verbindungslinien auf das Symbol Sicht ohne Verbindungen.

Klicken Sie zum Anzeigen der Server in der Domäne als Liste auf die Registerkarte Serverliste. Die Liste zeigt für jeden Server den Status, die Mitgliedsserver, den Typ, das übergeordnete Element (falls vorhanden), die IBM Informix-Version und das Datum und die Zeit an, zu der der Server zuletzt überwacht wurde.

Konfigurieren von Alertschwellenwerten

Konfigurieren Sie die Schwellenwerte für die Alerts für eine Enterprise Replication-Domäne (ER-Domäne).

Wenn ein Alertschwellenwert erreicht wird, wird eine Benachrichtigung an den Administrator gesendet und auf der Seite Domäne wird ein Alertsymbol angezeigt. Alertschwellenwerte gelten für alle Knoten in einer ER-Domäne.

Gehen Sie wie folgt vor, um Schwellenwerte zu konfigurieren:

  1. Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > ER-Domäne.
  2. Klicken Sie im Menü Aktionen auf Schwellenwerte editieren.
  3. Zum Aktivieren eines Schwellenwertes wählen Sie für den Schwellenwert das Kontrollkästchen Aktivieren aus. Zum Inaktivieren des Schwellenwerts löschen Sie den Inhalt des Kontrollkästchens.
  4. Geben Sie Schwellenwerte in das Feld ein oder ändern Sie sie.
  5. Klicken Sie auf Schwellenwerte aktualisieren, um Ihre Änderungen beizubehalten.

Prüfen auf Alerts in Enterprise Replication-Domäne

Prüfen Sie die Knoten in einer Enterprise Replication-Domäne auf Alerts.

Ein Alert tritt auf, wenn ein Knoten einen Alertschwellenwert erreicht oder überschreitet.

Gehen Sie wie folgt vor, um auf Alerts zu prüfen:

  1. Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > ER-Domäne.

    Alle Knoten mit Alerts werden hervorgehoben.

  2. Klicken Sie auf den Knoten und klicken Sie anschließend im Menü Aktionen auf Knotenstatus.

    Eine Seite zeigt die Profildaten und alle Alerts für den Knoten an.

  3. Wenn Sie die Replikationsaktivität für den ausgewählten Knoten überprüfen wollen, klicken Sie im Menü Aktionen auf Knotendetails.

    Die Seite Zusammenfassung zeigt allgemeine Statistikdaten für den ausgewählten Knoten an. Klicken Sie auf die entsprechenden Registerkarten, um ausführlichere Informationen über den Knoten anzuzeigen:

    • Erfassen: Datenerfassungsaktivität auf dem ausgewählten Knoten.
    • Sendewarteschlange: Die Sendewarteschlange für den ausgewählten Knoten und dessen Zielknoten und Replikate.
    • Netz: Die Netzaktivität zwischen dem ausgewählten Knoten und dessen Enterprise Replication-Partnerknoten.
    • Plattenbelegung: Die von dem ausgewählten Knoten verwendeten Speicherbereiche, wenn die Sende- und Empfangswarteschlangen sich füllen und Transaktionen auf Platte gespoolt werden.
    • Anwenden: Die am aktuellen Knoten angewendeten replizierten Transaktionen.
    • ATS: Die ATS-Dateien (Aborted Transactions Spooling) für den ausgewählten Knoten. Sie können fehlgeschlagene Transaktionen mithilfe der ATS-Dateien auf dieser Seite reparieren.
    • RIS: Die RIS-Dateien (Row Information Spooling) für den ausgewählten Knoten. Sie können fehlgeschlagene Transaktionen mithilfe der RIS-Dateien auf dieser Seite reparieren.
    • Fehler: Die auf den ausgewählten Knoten oder die Domäne bezogenen Fehler.
    • Konfiguration: Die für den ausgewählten Knoten definierten Enterprise Replication-Konfigurationsparameter. Sie können die Parameter über diese Seite bearbeiten.

Anzeigen des Status eines Enterprise Replication-Knotens

Zeigen Sie den Status eines Knotens in einer Enterprise Replication-Domäne (ER-Domäne) an.

Sie können den Status eines Knotens in einer Enterprise Replication-Domäne auf folgende Weise anzeigen:

  • Klicken Sie im OAT-Menü auf Replikation > ER-Domäne, um den Knoten anzuzeigen, mit dem IBM OpenAdmin Tool (OAT) for Informix verbunden ist. Die Detailseiten für den Knoten werden angezeigt.
  • Wenn Sie einen anderen Knoten in der Domäne anzeigen wollen, klicken Sie auf diesen Knoten. Klicken Sie anschließend im Menü Aktionen auf Knotenstatus. OAT stellt eine Verbindung mit dem ausgewählten Knoten in der Domäne her und die Detailseiten für den Knoten werden angezeigt.

Die Seite Zusammenfassung zeigt eine übergeordnete Zusammenfassung der Replikationsaktivität für den Knoten an, mit dem OAT verbunden ist. Klicken Sie zum Anzeigen detaillierter Informationen auf die Registerkarte des jeweiligen Abschnitts.

Anzeigen des Knotens als Replikationsquelle

Zeigen Sie einen Knoten in einer Enterprise Replication-Domäne (ER-Domäne) als Quellenserver an.

Enterprise Replication überwacht das logische Protokoll für Transaktionen, die auf Zielknoten repliziert werden sollen, und kopiert sie in die Sendewarteschlange auf dem Quellenserver.

Ausführliche Informationen zu den Transaktionen und zu der Sendewarteschlange können Sie auf diesen Seiten von IBM OpenAdmin Tool (OAT) for Informix anzeigen: Replikation > Knotendetails > Zusammenfassung, Erfassen und Sendewarteschlange.

Anzeigen des Knotens als Replikationsziel

Zeigen Sie einen Knoten in einer Enterprise Replication-Domäne (ER-Domäne) als Zielserver an.

Die Empfangswarteschlange ist der Wartebereich für Transaktionen, die auf den aktuellen Datenbankknoten angewendet werden sollen. Enterprise Replication wendet Transaktionen auf die richtigen Datenbanken an und benachrichtigt den Quellenknoten, wenn die Transaktion erfolgreich repliziert wurde.

Ausführliche Informationen zu den Transaktionen und zu der Sendewarteschlange können Sie auf diesen Seiten von IBM OpenAdmin Tool (OAT) for Informix anzeigen: Replikation > Knotendetails > Zusammenfassung, Sendewarteschlange, Anwenden, ATS und RIS.

Überwachen der Protokollerfassungsinformationen für einen Knoten

Sie können ausführliche Informationen zur Protokollerfassungsaktivität auf dem aktuellen Knoten in einer Enterprise Replication-Domäne (ER-Domäne) anzeigen.

Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > Knotendetails > Erfassen. Auf der Seite Erfassen werden die folgenden Informationen angezeigt:

  • Der Status der Enterprise Replication-Protokollerfassung auf dem aktuellen Knoten: Running (Aktiv), Down (Inaktiv) oder Uninitialized (Nicht initialisiert).
  • Die aktuelle Protokollposition und die Protokollposition, an der Transaktionen für die Replikation erfasst werden. Vergleichen Sie die Protokollpositionen, um zu ermitteln, ob Enterprise Replication mit den laufenden Datenbanktransaktionen Schritt hält.
  • Die Protokollwiederholungsposition befindet sich an der Stelle, ab der Enterprise Replication die Protokollinformationen nach einer Störung erneut in die Protokollaktualisierungspuffer einliest. Alle vor dieser Position generierten Transaktionen wurden von Enterprise Replication auf allen Zielservern angewendet oder im stabilen Warteschlangenbereich gespeichert.
    Wichtig:
    Stellen Sie zum Verhindern des Verlusts von Daten sicher, dass die Wiederholungsposition für das Protokoll nie überschrieben wird. Wenn die Wiederholungsposition sich nicht verändert, ist die Warteschlange möglicherweise voll oder ein ferner Knoten ist inaktiv.
  • Die Gesamtzahl der Protokollseiten und die Anzahl der Seiten, die verbleiben, bis Enterprise Replication in den DDRBLOCK-Modus wechselt, in dem Transaktionen blockiert werden.
  • Die Anzahl der Seiten, die bis zum Umlauf des logischen Protokolls und dem Überschreiben von Transaktionen verbleiben, die Enterprise Replication noch nicht verarbeitet hat.
  • Die Anzahl der Protokollseiten, die während der Protokollerfassung gelesen wurden, entweder aus dem Cache für die Protokollerfassung oder von gespoolten Protokolleinträgen auf Platte, und ob Überläufe des Protokollcache vorliegen.
  • Ob dynamisches Protokollieren aktiviert ist, um den DDRBLOCK-Modus zu verhindern, sowie zugehörige Informationen, einschließlich der Anzahl dynamischer logischer Protokolle, die angefordert werden können, und der Anzahl der Anforderungen zur Erstellung dynamischer logischer Protokolle.

Verhindern von DDRBLOCK und Protokollumlauf

Verhindern Sie, dass Enterprise Replication (ER) in den DDRBLOCK-Modus wechselt, in dem Transaktionen blockiert werden.

Bei einem Umlauf des logischen Protokolls werden die Transaktionen überschrieben, die Enterprise Replication noch nicht verarbeitet hat. Die Transaktionen gehen verloren, bevor sie in der Sendewarteschlange gespeichert werden. Steht ein Protokollumlauf für einen Datenbankserver unmittelbar bevor, wechselt Enterprise Replication in den DDRBLOCK-Modus. Enterprise Replication kann so konfiguriert werden, dass nicht in den DDRBLOCK-Modus gewechselt wird, sondern dynamisch eine Datei für ein logisches Protokoll hinzugefügt wird.

  • Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > Knotendetails > Erfassen, um die Nähe zu DDRBLOCK zu überwachen. Prüfen Sie das Feld Nähe zu DDRBLOCK, das die Anzahl der Seiten im Speicherbereich für logische Protokolle anzeigt, die verbleiben, bevor Enterprise Replication in den DDRBLOCK-Modus wechselt.
  • Wenn der DDRBLOCK-Modus verhindert werden soll, müssen Sie Enterprise Replication so konfigurieren, dass Dateien für logische Protokolle dynamisch hinzugefügt werden. Klicken Sie im OAT-Menü auf Replikation > Knotendetails > Konfigurieren und legen Sie die Parameter DYNAMIC_LOGS und CDR_MAX_DYNAMIC_LOGS fest.
  • Klicken Sie im OAT-Menü auf Replikation > Knotendetails > Erfassen, um die Nähe zum Protokollumlauf zu überwachen. Prüfen Sie das Feld Nähe zu Protokollumlauf, das die Anzahl der Seiten im Speicherbereich für logische Protokolle anzeigt, die bis zum Umlauf des logischen Protokolls verbleiben.

  • Handhabung potenzieller Protokollumläufe (Enterprise Replication - Handbuch)

Aktivieren der dynamischen Zuordnung von Protokolldateien

Verhindern Sie, dass Enterprise Replication (ER) in den DDRBLOCK-Modus wechselt, in dem Transaktionen blockiert werden, indem Sie Enterprise Replication so konfigurieren, dass Dateien für logische Protokolle dynamisch hinzugefügt werden.

Bei einem Umlauf des logischen Protokolls werden die Transaktionen überschrieben, die Enterprise Replication noch nicht verarbeitet hat. Die Transaktionen gehen verloren, bevor sie in der Sendewarteschlange gespeichert werden. Steht ein Protokollumlauf für einen Datenbankserver unmittelbar bevor, wechselt Enterprise Replication in den DDRBLOCK-Modus. Enterprise Replication kann so konfiguriert werden, dass nicht in den DDRBLOCK-Modus gewechselt wird, sondern dynamisch eine Datei für ein logisches Protokoll hinzugefügt wird.

Gehen Sie wie folgt vor, um die dynamische Protokolldateizuordnung zu aktivieren:

  1. Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > Knotendetails > Konfigurieren.
  2. Klicken Sie auf Konfiguration editieren.
  3. Geben Sie für DYNAMIC_LOGS den Wert 2 ein, um die dynamische Protokolldateizuordnung zu aktivieren.
  4. Geben Sie für CDR_MAX_DYNAMIC_LOGS die Anzahl der dynamischen Protokolldateianforderungen ein, die Enterprise Replication in einer einzelnen Serversitzung stellen kann. Sie können auch -1 eingeben, wenn unbegrenzt Protokolle hinzugefügt werden sollen.

  • Zugehörige Konzepte:
  • Handhabung potenzieller Protokollumläufe (Enterprise Replication - Handbuch)

Überwachen der Sendewarteschlange für einen Knoten

Überwachen Sie Statistiken zu der Sendewarteschlange auf dem aktuellen Knoten in einer Enterprise Replication-Domäne (ER-Domäne) sowie zu den Zielknoten und Replikaten, für die der aktuelle Knoten eine Quelle replizierter Transaktionen ist.

Enterprise Replication erfasst Informationen aus den logischen Protokollen und stellt die zu übertragenden Daten in die Sendewarteschlange. Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > Knotendetails > Sendewarteschlange, um die Sendewarteschlange zu überwachen.

Die Seite Sendewarteschlange zeigt Statistikdaten zu den Transaktionen in der Sendewarteschlange an. Sie stellt Informationen zum Speicherbereich, zu den an Zielknoten gesendeten Transaktionen und zu dem für Transaktionen in der Sendewarteschlange erforderlichen Speicher bereit. Darüber hinaus enthält sie Informationen zu den Zielknoten und Replikaten, für die der aktuelle Knoten eine Quelle replizierter Transaktionen ist.

  • Prüfen Sie die Felder Transaktionen in Warteschlange, Transaktionen im Speicher und Gespoolte Transaktionen, um zu ermitteln, wie viele Transaktionen sich in der Warteschlange befinden und ob sie gespeichert werden.
  • Prüfen Sie das Feld Protokollerfassungsposition, um zu ermitteln, ob Transaktionen auf dem Zielknoten angewendet werden. Vergleichen Sie es mit dem Feld Zuletzt bestätigte Daten, das die Position der letzten Transaktion, die auf dem Zielknoten angewendet wurde, im logischen Protokoll angibt.
  • Prüfen Sie das Feld Ausgesetzte Knoten, um zu ermitteln, ob einer der Zielknoten Transaktionen nicht anwendet. Klicken Sie auch auf die Registerkarte Zielknoten, um den Verbindungsstatus der einzelnen zugehörigen Zielknoten zu überprüfen.
  • Klicken Sie auf die Registerkarte Zielknoten, um den Status der Zielknoten anzuzeigen, an die der aktuelle Knoten Daten repliziert.
  • Klicken Sie auf die Registerkarte Replikate, um die Replikate des aktuellen Knoten anzuzeigen.

Definieren eines Replikatsets

Erstellen Sie ein neues Replikatset.

Ein Replikatset kombiniert mehrere Replikate und bildet dadurch ein Set, das zusammen als eine Einheit verwaltet werden kann. Sie können mit einem einzigen Befehl alle Replikate im Set starten, stoppen, aussetzen oder wiederaufnehmen.

Gehen Sie wie folgt vor, um ein Replikatset zu definieren:

  1. Erweitern Sie Replikation im Menü von IBM OpenAdmin Tool (OAT) for Informix und klicken Sie dann auf Replikate.
  2. Klicken Sie auf die Registerkarte Replikatsets.
  3. Klicken Sie im Menü Aktionen auf Replikatset definieren.
  4. Geben Sie einen Namen für das Replikatset ein. Der Name muss eindeutig sein und darf nicht mit einem Replikatnamen identisch sein.
  5. Wählen Sie die Replikate für das Set aus.
    • Wenn Sie Replikate hinzufügen wollen, wählen Sie diese in der Liste Verfügbare Replikate aus und klicken Sie anschließend auf Hinzufügen oder auf Alle hinzufügen.
    • Wenn Sie Replikate entfernen wollen, wählen Sie diese in der Liste Dieses Replikatset aus und klicken Sie anschließend auf Entfernen oder klicken Sie auf Alle entfernen.
  6. Optional: Damit dieses Set ein exklusives Replikatset wird, wählen Sie Replikate sind exklusiv für dieses Set aus.
  7. Klicken Sie zum Festlegen der Frequenzoptionen für das Replikatset auf Weiter oder klicken Sie auf Fertigstellen, um das Replikatset mit dem Standardwert zu erstellen. .
  8. Optional: Geben Sie die Frequenz der Replikation an. Standardmäßig erfolgt die Replikation unverzüglich nach dem Eintreffen der Daten.
  9. Klicken Sie auf Fertigstellen, um das Replikatset zu erstellen.

  • Zugehörige Tasks:
  • Exklusive Replikatsets (Enterprise Replication - Handbuch)

Definieren eines Replikats

Definieren Sie die Replikationsteilnehmer und die Replikationsattribute einschließlich der Häufigkeit, mit der die Daten repliziert werden sollen, und der Art und Weise, wie Konflikte zu beheben sind.

Mit der Aktion zum Definieren von Replikaten können Sie ein strenges Masterreplikat und seine Teilnehmer definieren. Über ein Masterreplikat wird sichergestellt, dass replizierte Tabellen auf unterschiedlichen Servern konsistente Spaltendatentypen aufweisen. Ein strenges Masterreplikat stellt außerdem sicher, dass die replizierten Tabellen identische Spaltennamen aufweisen. Alle Datenbankserver mit Masterreplikaten müssen eine Direktverbindung zum Datenbankserver, auf dem das Masterreplikat basiert, herstellen können.

Voraussetzung:
Eine Tabelle muss einen Primärschlüssel aufweisen, um Replikationsteilnehmer sein zu können. Darüber hinaus muss es sich dabei tatsächlich um einen Tabellennamen handeln. Ein Synonym, der Name einer Sicht oder einer Sequenz ist nicht zulässig. Es darf sich auch nicht um eine externe Tabelle handeln. Nur Tabellen, die Replikationsteilnehmer sein können, werden in der Liste der auszuwählenden Tabellen angezeigt.

Gehen Sie wie folgt vor, um ein Replikat und die Replikationsteilnehmer zu definieren:

  1. Erweitern Sie Replikation im Menü von IBM OpenAdmin Tool (OAT) for Informix und klicken Sie dann auf Replikate.
  2. Klicken Sie auf die Registerkarte Replikate.
  3. Klicken Sie im Menü Aktionen auf Replikat definieren.
  4. Geben Sie einen eindeutigen Namen für das Replikat ein.
  5. Wählen Sie die Servergruppe, die als Basis für das Masterreplikat verwendet werden soll, aus der Liste Masterreplikatserver aus.
  6. Wenn die Tabellen, die auf dem Masterreplikatserver, aber nicht auf dem Zielserver vorhanden sind, automatisch erstellt werden sollen, wählen Sie Fehlende Tabellen auf Zielservern erstellen aus.
  7. Wählen Sie mithilfe der Filter die Tabellen und Replikationsserver aus, die Sie als Teilnehmer im Replikat definieren wollen.
    1. Geben Sie als Datenbankfilter den Datenbanknamen vollständig oder teilweise ein.
    2. Geben Sie als Tabellenfilter den Tabellennamen vollständig oder teilweise ein. Fahren Sie mit dem Filtern der Liste fort, bis für jeden Server ein einzelner Teilnehmer angezeigt wird.
    3. Klicken Sie auf Auswählen, um alle von den Filtern angegebenen Teilnehmer auszuwählen.

      Sie können auch eine Tabelle aus- oder abwählen, indem Sie in der Liste auf die Tabelle klicken.

    Hinweis:
    Der Masterreplikatserver muss ein Teilnehmer im neuen Replikat sein. Wenn Sie die Tabellen auswählen, die Teilnehmer im Replikat sein sollen, müssen Sie eine Tabelle auf dem Masterreplikatserver auswählen.
  8. Geben Sie den Teilnehmertyp an: Nur Empfänger oder Primär. Die Standardeinstellung lautet Primär: Der Teilnehmer sendet und empfängt Daten. Klicken Sie auf das Symbol, um die Option zu ändern.
  9. Geben Sie an, welche Zugriffsrechte zum Anwenden der replizierten Daten verwendet werden. Die Standardeinstellung ist der Tabelleneigner. Klicken Sie auf das Symbol, um die Option zu ändern.
  10. Klicken Sie zum Festlegen der erweiterten Optionen für das Replikat auf Weiter oder klicken Sie auf Fertigstellen, um das Replikat mit den Standardwerten für die erweiterten Optionen zu erstellen.

    Enterprise Replication überprüft, ob die Tabellen auf den verschiedenen Servern konsistente Spaltendatentypen und identische Spaltennamen aufweisen. Wenn die Optionen Weiter und Fertigstellen nicht verfügbar sind, sind die Tabellen nicht konsistent.

  11. Optional: Geben Sie die erweiterten Optionen an:
    • Konfliktlösungsbereich. Standardeinstellung: Transaktion.
    • Konfliktlösungsregel. Standardeinstellung: Zeitmarke.
    • Gleitkommaformat. Standardeinstellung: IEEE.
    • Spoolen abgebrochener Transaktionen aktivieren. Standardmäßig ist ATS nicht aktiviert.
    • Spoolen von Zeileninformationen aktivieren. Standardmäßig ist RIS nicht aktiviert.
    • Vollständige Zeile replizieren. Standardmäßig werden vollständige Zeilen repliziert. Diese Option ist aktiviert.
    • Trigger am Ziel auslösen. Standardmäßig werden Trigger in der Zieltabelle nicht ausgelöst.
    • Gelöschte Zeilen beibehalten. Standardmäßig werden gelöschte Zeilen nicht beibehalten.
    • Frequenz der Replikation. Standardmäßig erfolgt die Replikation unverzüglich nach dem Eintreffen der Daten.
  12. Klicken Sie auf Fertigstellen, um das Replikat zu erstellen.

Eine Vorlage definiert eine Gruppe von Masterreplikaten und ein Replikatset. Eine Vorlage besteht aus Schemainformationen zu einer Datenbank, einer Gruppe von Tabellen, Spaltenattributen und den Primärschlüsseln, die Zeilen angeben.

Gehen Sie folgt vor, um eine Vorlage zu definieren:

  5. Erweitern Sie in der Liste Server die Servernamen und die Datenbanken, um die Tabellen anzuzeigen.
    Tipp:
    Beginnen Sie zum Filtern der Liste, den Servernamen in das Textfeld Serverfilter und den Datenbanknamen in das Textfeld Datenbankfilter einzugeben.
  6. Wählen Sie die Tabellen für die Vorlage aus.
    • Wählen Sie die Datenbank in der Liste Tabellen aus und klicken Sie anschließend auf Hinzufügen, um alle Tabellen in einer Datenbank zur Liste hinzuzufügen.

    • Konfliktlösungsbereich. Standardeinstellung: Transaktion.
    • Konfliktlösungsregel. Standardeinstellung: Ignorieren.
    • Gleitkommaformat. Standardeinstellung: IEEE.
    • Spoolen abgebrochener Transaktionen aktivieren. Standardmäßig ist ATS nicht aktiviert.
    • Spoolen von Zeileninformationen aktivieren. Standardmäßig ist RIS nicht aktiviert.
    • Trigger am Ziel auslösen. Standardmäßig werden Trigger in der Zieltabelle nicht ausgelöst.
    • Gelöschte Zeilen beibehalten. Standardmäßig werden gelöschte Zeilen nicht beibehalten.

Sie können nun die Vorlage auf Datenbankserver in der ER-Domäne instanziieren, indem Sie die Aktion Vorlage realisieren verwenden.

Voraussetzung:

  3. Wählen Sie in der Liste der Vorlagen die Vorlage aus, die instanziiert werden soll.
  4. Klicken Sie im Menü Aktionen auf Vorlage realisieren.
  5. Wählen Sie in der Liste Verfügbare Server die Server aus, auf denen die Vorlage instanziiert werden soll.
    Tipp:
    Beginnen Sie, den Servernamen im Textfeld Serverfilter einzugeben, um die Liste zu filtern.
  6. Zum Erstellen der Vorlage auf den Servern anhand der Standardoptionen klicken Sie auf Fertigstellen oder klicken Sie auf Weiter, um eine beliebige der folgenden Optionen anzugeben. Klicken Sie anschließend auf Fertigstellen.
    • Fehlende Tabellen auf Zielservern erstellen. Wenn diese Option ausgewählt ist, werden Tabellen, die auf dem Masterreplikatserver, jedoch nicht auf den Zielservern vorhanden sind, automatisch auf den Zielservern erstellt. Diese Option ist standardmäßig ausgewählt. Der Datenbankbereich für die erstellten Tabellen wird standardmäßig auf denselben Datenbankbereich wie die Datenbank gesetzt.

    • Nur Empfänger. Zielserver als Empfängerserver angeben. Standardmäßig senden die Server replizierte Daten und empfangen sie auch.
    • Daten synchronisieren. Geben Sie einen Server an, der für die Datensynchronisation auf den Servern, auf denen die Vorlage instanziiert wird, als Quelle agiert. Geben Sie an, wie auf dem Datenquellenserver nicht vorhandene Zeilen auf den Zielservern gehandhabt werden sollen. Standardmäßig werden zusätzliche Zeilen von den Zielservern gelöscht.

Erstellen eines Service-Level-Agreements für ein Raster

Erstellen Sie ein Service-Level-Agreement (SLA) für ein Raster mithilfe einer Connection Manager-Instanz.

Voraussetzungen:
Für das Raster erfordert IBM Informix-Plug-in 'Replication' für OpenAdmin Tool (OAT) IBM Informix 11.70. OpenAdmin Tool (OAT) for Informix erfordert für Connection Manager IBM Informix 11.70.xC3 und den zum Lieferumfang von Informix 11.70.xC3 oder IBM Informix Client Software Development Kit (Client SDK) 3.70.xC3 gehörenden Connection Manager. Das Raster muss als Verbindungseinheit für eine Connection Manager-Instanz definiert sein.

Ein SLA (Service-Level-Agreement) ist ein Vertrag zwischen angegebenen Clientanwendungen und deren Service-Providern, in diesem Fall Informix-Datenbankservern. Connection Manager stellt die Verbindungen zwischen den Clientanwendungen und den entsprechenden Servern auf der Grundlage der Service-Level-Agreements her. Connection Manager akzeptiert Clientverbindungsanforderungen und leitet die Anforderungen entweder an den entsprechenden Server um oder fungiert als Proxy-Server zwischen den Clientanwendungen und den Servern.

Wenn die Datenqualität überwacht wird, kann die Connection Manager-Instanz Clientverbindungen auf der Basis der Anzahl Anwendungsfehler, der Transaktionslatenzzeit und der Workload an die Replikationsserver weiterleiten. Wenn die Datenqualität nicht überwacht wird, kann die Connection Manager-Instanz die Clientverbindungen nur auf der Basis der Workload weiterleiten. Zur Angabe der Anwendungsfehler und der Latenzzeit in einem Service-Level-Agreement aktivieren Sie die Überwachung der Datenqualität, indem Sie auf der Seite Replication > ER-Domäne einen Server modifizieren, der ein Mitglied des Rasters ist.

Gehen Sie wie folgt vor, um ein Service-Level-Agreement für ein Raster zu erstellen:

  1. Klicken Sie im OAT-Menü auf Replikation > Connection Manager.
  2. Klicken Sie auf die Connection Manager-Instanz.
  3. Klicken Sie auf das Raster in der Liste.
  4. Klicken Sie im Menü Aktionen auf SLA erstellen.
  5. Beenden Sie den Assistenten.

Verwalten von Enterprise Replication-Servern

Modifizieren Sie die Attribute eines Enterprise Replication-Servers (ER) oder ändern Sie den Status des ER-Servers.

Modifizieren eines ER-Servers

Modifizieren Sie einen Enterprise Replication-Server (ER), um das ATS- oder das RIS-Verzeichnis, das Inaktivitätszeitlimit oder den Typ einiger Teilnehmer auf dem Server zu ändern.

Ändern der Position und des Formats von ATS- und RIS-Dateien

Modifizieren Sie einen Enterprise Replication-Server (ER), um die Position oder das Dateiformat der Dateien zum Spoolen abgebrochener Transaktionen (ATS) und der Dateien zum Spoolen von Zeileninformationen (RIS) zu ändern.

Voraussetzung:
Erstellen Sie die Position für das Verzeichnis, bevor Sie das ATS- oder das RIS-Verzeichnis ändern.

Gehen Sie wie folgt vor, um die Position und das Format der ATS- und der RIS-Dateien zu ändern:

  1. Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > ER-Domäne.
  2. Wählen Sie den Server auf der Seite Routing-Topologie oder Serverliste aus.
  3. Klicken Sie im Menü Aktionen auf Server modifizieren.
  4. Wählen Sie ATS-Dateien in folgendem Verzeichnis generieren aus und geben Sie den vollständigen Pfad eines vorhandenen Verzeichnisses ein, in dem die ATS-Dateien gespeichert werden sollen.
  5. Wählen Sie RIS-Dateien in folgendem Verzeichnis generieren aus und geben Sie den vollständigen Pfad eines vorhandenen Verzeichnisses ein, in dem die RIS-Dateien gespeichert werden sollen.
  6. Geben Sie das Format für die ATS- und RIS-Dateien an:
    • Wählen Sie Text aus, um Textdateien zu generieren, die Enterprise Replication verarbeiten kann, um fehlgeschlagene Transaktionen zu reparieren. (Standardeinstellung)
    • Wählen Sie XML aus, um XML-Dateien zu generieren, die Sie beim Schreiben benutzerdefinierter Reparaturscripts verwenden können.
    • Wählen Sie Beide aus, um Textdateien und XML-Dateien zu generieren.
  7. Klicken Sie auf Speichern.

    Die von Ihnen ausgewählten Optionen und die Ergebnisse der Aktion werden angezeigt.

    Sie können die Generierung von ATS- und RIS-Dateien für ein Replikat über die Aktion Replikat modifizieren auf der Seite Replikate aktivieren.

Ändern des Inaktivitätszeitlimits für einen ER-Server

Modifizieren Sie einen Enterprise Replication-Server (ER), um das Inaktivitätszeitlimit zu ändern.

Das Inaktivitätszeitlimit ist die Zeitspanne, die die Verbindung zwischen dem Server und den anderen Servern in der ER-Domäne inaktiv sein kann, bevor der Server die Verbindung trennt.

Gehen Sie wie folgt vor, um das Inaktivitätszeitlimit für einen ER-Server zu ändern:

  1. Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > ER-Domäne.
  2. Wählen Sie den Server auf der Seite Routing-Topologie oder Serverliste aus.
  3. Klicken Sie im Menü Aktionen auf Server modifizieren.
  4. Wählen Sie für das Inaktivitätszeitlimit die Option Ein aus und geben Sie die Zeit in Minuten an.

    Die Eingabe 0 (null) Minuten oder das Freilassen des Felds hat den gleichen Effekt wie die Auswahl von Aus.

  5. Klicken Sie auf Speichern.

    Die von Ihnen ausgewählten Optionen und die Ergebnisse der Aktion werden angezeigt.

Ändern des Typs für Teilnehmer auf einem ER-Server

Modifizieren Sie einen Enterprise Replication-Server (ER), um den Typ für Teilnehmer zu ändern, deren Konfliktlösungsregel ignorieren ist.

Der Teilnehmertyp gibt an, ob ein Teilnehmer replizierte Daten nur empfängt oder ob er sie empfängt und sendet. Der Teilnehmertyp hat keinen Einfluss auf die Konfliktlösung, wenn die Regel ignorieren lautet. Daher können alle Teilnehmer auf einem Server in Replikaten mit dieser Regel mit einer einzigen Aktion geändert werden.

Gehen Sie wie folgt vor, um den Typ für Teilnehmer zu ändern:

  1. Klicken Sie im Menü von IBM OpenAdmin Tool (OAT) for Informix auf Replikation > ER-Domäne.
  2. Wählen Sie den Server auf der Seite Routing-Topologie oder Serverliste aus.
  3. Klicken Sie im Menü Aktionen auf Server modifizieren.
  4. Wählen Sie den Teilnehmertyp aus:
    • Primär: Die Teilnehmer senden und empfangen replizierte Daten.
    • Nur Empfänger: Die Teilnehmer empfangen nur Daten von Primärteilnehmern.
  5. Klicken Sie auf Speichern.

    Die von Ihnen ausgewählten Optionen und die Ergebnisse der Aktion werden angezeigt.

    Tipp:
    Sie können darüber hinaus den Typ für diese Replikate und für andere Replikate auf der Seite Replikate mit der Aktion Replikat modifizieren ändern.

Das Raster, der Cluster oder das Replikatset muss als Verbindungseinheit für eine Connection Manager-Instanz definiert sein.

Ein SLA (Service-Level-Agreement) ist ein Vertrag zwischen angegebenen Clientanwendungen und deren Service-Providern, in diesem Fall Informix-Datenbankservern. Connection Manager stellt die Verbindungen zwischen den Clientanwendungen und den entsprechenden Servern auf der Grundlage der Service-Level-Agreements her. Connection Manager akzeptiert Clientverbindungsanforderungen und leitet die Anforderungen entweder an den entsprechenden Server um oder fungiert als Proxy-Server zwischen den Clientanwendungen und den Servern.

Connection Manager bietet einen Lastausgleich und leitet Clientverbindungsanforderungen an den entsprechenden Server in einer Verbindungseinheit auf der Basis von Service-Level-Agreements (SLAs) weiter. Wenn die Datenqualität überwacht wird, kann die Connection Manager-Instanz Clientverbindungen auf der Basis der Anzahl Anwendungsfehler, der Transaktionslatenzzeit und der Workload an die Replikationsserver weiterleiten. Wenn die Datenqualität nicht überwacht wird, kann die Connection Manager-Instanz die Clientverbindungen nur auf der Basis der Workload weiterleiten. Zur Angabe der Anwendungsfehler und der Latenzzeit in einem Service-Level-Agreement aktivieren Sie die Überwachung der Datenqualität, indem Sie auf der Seite Replication > ER-Domäne einen Server modifizieren, der ein Mitglied des Rasters ist.

  1. Klicken Sie im OAT-Menü auf Replikation > Connection Manager.
  2. Klicken Sie auf die Connection Manager-Instanz.
  3. Klicken Sie auf die Verbindungseinheit in der Liste.
  4. Klicken Sie im Menü Aktionen auf SLA erstellen.
  5. Beenden Sie den Assistenten.
