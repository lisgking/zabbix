Informix Replication Plug-in for OAT

O IBM® Informix® Replication Plug-in for OpenAdmin Tool (OAT) fornece uma interface gráfica para configurar, monitorar e administrar o Enterprise Replication (ER). É possível também administrar grades de servidores de replicação interconectados.

Pré-requisitos:
O IBM Informix 11.50.xC4 ou mais recente é necessário para o plug-in Replication. Para o Gerenciador de Conexões, o IBM OpenAdmin Tool (OAT) for Informix requer o Informix 11.70.xC3 e o Gerenciador de Conexões que está empacotado com o Informix 11.70.xC3 ou com o IBM Informix Client Software Development Kit (Client SDK) 3.70.xC3

Com o plug-in Replication, é possível monitorar servidores e domínios do ER:

  • Visualizar uma topologia do domínio ER e todos os nós conhecidos.
  • Visualizar informações resumidas e detalhadas sobre os nós no domínio.
  • Visualizar informações detalhadas sobre as réplicas e conjuntos de réplicas no domínio.
  • Configurar limites de alerta para o domínio.

É possível concluir tarefas administrativas do ER:

  • Definir e modificar servidores ER, réplicas e conjuntos de réplicas.
  • Usar modelos para configurar o ER.
  • Gerenciar o estado dos servidores, conjuntos de réplicas e réplicas.
  • Ressincronizar dados entre os servidores de replicação.
  • Alterar os parâmetros de configuração do ER para um servidor.

É possível criar e administrar grades de servidores de replicação interconectados em um domínio:

  • Visualizar informações sobre as grades em um domínio.
  • Criar uma grade e ativar servidores de origem e usuários para executarem comandos de grade.
  • Modificar uma grade para incluir servidores, ativar ou desativar servidores de origem ou autorizar usuários adicionais.
  • Revisar o status de comandos de grade e executar novamente os comandos de grade que falharem.
  • Rotear conexões do cliente para servidores na grade com base na qualidade de dados replicados, configurando Acordos de Nível de Serviço (SLAs) para um Gerenciador de Conexões.

É possível definir um Gerenciador de Conexões para gerenciar solicitações de conexão do cliente para uma combinação de clusters, grades, conjuntos de replicações e conjuntos de servidores. É possível incluir unidades de conexões em um Gerenciador de Conexões, iniciar e parar um Gerenciador de Conexões e criar Acordos de Nível de Serviço (SLAs).

Configurando o Plug-in Replication

Configure o IBM Informix Replication Plug-in for OpenAdmin Tool (OAT) para monitorar e administrar o Enterprise Replication (ER).

Para configurar o plug-in Replication:

  1. Instale o plug-in Replication.
  2. Crie grupos do OAT.
  3. Efetue logon no OAT com um dos servidores em um domínio existente do ER ou qualquer servidor que você desejar definir para o Enterprise Replication.

    Para obter informações adicionais sobre como instalar o plug-in Replication e criar grupos do OAT, consulte a ajuda do OAT. Para acessar a ajuda, no menu do OAT, clique em Ajuda > Ajuda da Tarefa OAT.

  • Configurando e Gerenciando o Enterprise Replication (Guia do Enterprise Replication)

Monitorando o Enterprise Replication

Use o IBM Informix Replication Plug-in for OpenAdmin Tool (OAT) para monitorar servidores e domínios do Enterprise Replication.

Visualizando a Topologia de um Domínio do ER

Visualize a topologia, ou mapa de roteamento, de um domínio do Enterprise Replication (ER).

Para visualizar a topologia do domínio atual, no menu para o IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Domínio do ER.

A página Domínio exibe todos os nós do domínio ER que são conhecidos pelo nó atual; por exemplo, um nó folha conhece seu pai e a si próprio. Nós são denotados pelo nome de grupo definido no arquivo SQLHOSTS.

Importante:
Se você estiver conectado a um nó não-raiz ou a um nó folha, as informações que poderá visualizar e as ações que poderá tomar serão limitadas. Para acesso total, vá para um nó-raiz no domínio.

Na página Domínio, você pode:

  • Determinar se algum nó precisa de atenção revisando os alertas visuais.
  • Ver o nome do nó e qualquer membro dele passando o mouse sobre o ícone de nó.
  • Exibir ou ocultar a subárvore de algum nó não-folha clicando no ícone de expansão.
  • Customizar sua visualização:
    • Para aumentar ou diminuir o zoom no domínio, use a régua de controle Zoom.
    • Para visualizar a taxa de atualização, clique no ícone de relógio.
    • Para alterar a taxa de atualização, clique no ícone de relógio e use a régua de controle para especificar uma taxa de atualização de 0 a 300 segundos.
    • Para exibir as linhas de conexão entre os nós, clique no ícone Visualização Normal.
    • Para ocultar as linhas de conexão, clique no ícone Nenhuma Visualização de Conexão.

Para visualizar os servidores do domínio em uma lista, clique na guia Lista de Servidores. Para cada servidor, a lista exibe o status, os servidores de membros, o tipo, o pai (se aplicável), a versão do IBM Informix e a data e hora em que o servidor foi monitorado pela última vez.

Configurando Limites de Alerta

Configure os limites para os alertas de um domínio do Enterprise Replication (ER).

Quando um limite de alerta é atingido, uma notificação é enviada para o administrador e um ícone de alerta aparece na página Domínio. Limites de alerta aplicam-se a todos os nós em um domínio ER.

Para configurar limites:

  1. No menu para o IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Domínio do ER.
  2. No menu Ações, clique em Editar Limites.
  3. Para ativar um limite, selecione a caixa de opção Ativar para o limite. Para desativar o limite, desmarque a caixa de opção.
  4. Insira ou altere valores de limite no campo.
  5. Clique em Atualizar Limites para manter suas alterações.

Verificando Alertas em um Domínio Enterprise Replication

Verifique alertas nos nós em um domínio de Enterprise Replication (ER).

Um alerta surge quando um nó atinge ou excede um limite de alerta.

Para verificar alertas:

  1. No menu para o IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Domínio do ER.

    Todos os nós com alertas são destacados.

  2. Clique no nó e, então, no menu Ações, clique em Status do Nó.

    Uma página exibe os dados do perfil e todos os alertas do nó.

  3. Para revisar a atividade de replicação do nó selecionado, no menu Ações, clique em Detalhes do Nó.

    A página Resumo exibe estatísticas de alto nível para o nó selecionado. Clique nas guias para obter informações detalhadas sobre o nó:

    • Captura: A atividade de captura de dados no nó selecionado.
    • Fila de Envio: A fila de envio do nó selecionado e seus nós de destino e replicações.
    • Rede: A atividade de rede entre o nó selecionado e seus nós ER de parceiros.
    • Uso do Disco: Os espaços usados pelo nó selecionado quando as filas de envio e recebimento são preenchidas e as transações são colocadas em spool no disco.
    • Aplicar: As transações replicadas são aplicadas no nó atual.
    • ATS: Os arquivos Aborted Transaction Spooling (ATS) do nó selecionado. Você pode reparar transações com falha usando os arquivos ATS desta página.
    • RIS: Os arquivos Row Information Spooling (RIS) para o nó selecionado. Você pode reparar transações com falha usando os arquivos RIS desta página.
    • Erros: Os erros relacionados ao nó ou ao domínio selecionado.
    • Configuração: Os parâmetros de configuração de ER do nó selecionado. Você pode editar os parâmetros desta página.

Visualizando o Status de um Nó do Enterprise Replication

Visualiza o status de um nó em um domínio Enterprise Replication (ER).

Você pode visualizar o status de um nó em um domínio ER das seguintes maneiras:

  • Para visualizar o nó ao qual o IBM OpenAdmin Tool (OAT) for Informix está conectado, no menu do OAT, clique em Replicação > Domínio do ER. As páginas detalhadas do nó serão exibidas.
  • Para visualizar um no diferente no domínio, clique no nó. Em seguida, no menu Ações, clique em Status do Nó. O OAT se conecta ao nó selecionado no domínio e as páginas detalhadas do nó são exibidas.

Monitorando Informações de Captura de Log para um Nó

Visualize informações detalhadas sobre a atividade de captura de log no nó atual em um domínio do Enterprise Replication (ER).

No menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Capturar. A página Capturar exibe as informações a seguir:

  • O estado da captura de log do ER no nó atual: Em Execução, Inativo ou Não Inicializado.
  • A posição do log atual e a posição do log no qual as transações estão sendo capturadas para replicação. Compare as posições do log para determinar se o ER está mantendo-se ativo nas transações em andamento do banco de dados.
  • A posição de reprodução do log, que é o ponto a partir do qual o ER inicia a releitura das informações de log nos buffers de atualização de log após uma falha. Todas as transações geradas antes dessa posição em todos os servidores de destino foram aplicadas pelo ER ou armazenadas de modo estável no espaço na fila.
    Importante:
    Para evitar perda de dados, assegure-se de que a posição de reprodução do log nunca seja sobrescrita. Se a posição de reprodução não estiver avançando, a fila poderá estar cheia ou um nó remoto poderá estar inativo.
  • O número total de páginas de log e o número de páginas até que o ER entre em um estado de bloqueio de transação, conhecido como modo DDRBLOCK.
  • O número de páginas até que o log lógico quebre e substitua as transações ainda não processadas pelo ER.
  • O número de páginas de log que foram lidas durante a captura do log, a partir do cache de captura de log e das entradas de log em spool no disco, e se houver estouro do cache de log.
  • Se a criação de log dinâmica estiver ativada para evitar o modo DDRBLOCK e informações relacionadas, incluindo o número de logs lógicos dinâmicos que podem ser solicitados e o número de pedidos feitos para criar logs lógicos dinâmicos.

Evitando DDRBLOCK e uma Quebra de Log

Evite que o Enterprise Replication (ER) entre em um estado de bloqueio da transação, conhecido como modo DDRBLOCK.

Quando um log lógico quebra, ele substitui as transações ainda não processadas pelo ER. As transações são perdidas antes que sejam armazenadas na fila de envio. Se um servidor de banco de dados estiver abordando a quebra do log, o ER entra no modo DDRBLOCK. O ER pode ser configurado para incluir um arquivo de log lógico dinamicamente em vez de entrar no modo DDRBLOCK.

  • Para monitorar a proximidade para DDRBLOCK, no menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Capturar. Revise o campo Proximidade com DDRBLOCK,que exibe o número de páginas no espaço do log lógico até que o ER entre em DDRBLOCK.
  • Para evitar o DDRBLOCK, configure o ER para incluir arquivos de log lógico dinamicamente. No menu do OAT, clique em Replicação > Detalhes do Nó > Configurar e configure os parâmetros DYNAMIC_LOGS e CDR_MAX_DYNAMIC_LOGS.
  • Para monitorar a proximidade para agrupamento de log, no menu do OAT, clique em Replicação > Detalhes do Nó > Capturar. Revise o campo Proximidade com Quebra do Log, que exibe o número de páginas no espaço do log lógico até a quebra do log lógico.

Ativando a Alocação do Arquivo de Log Dinâmico

Ajuda a evitar que o Enterprise Replication (ER) entre em um estado de bloqueio da transação, conhecido como modo DDRBLOCK, configurando o ER para incluir arquivos de log lógicos dinamicamente.

Quando um log lógico quebra, ele substitui as transações ainda não processadas pelo ER. As transações são perdidas antes que sejam armazenadas na fila de envio. Se um servidor de banco de dados estiver abordando a quebra do log, o ER entra no modo DDRBLOCK. O ER pode ser configurado para incluir um arquivo de log lógico dinamicamente em vez de entrar no modo DDRBLOCK.

Para ativar a alocação do arquivo de log dinâmico:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Configurar.
  2. Clique em Editar Configuração.
  3. Para DYNAMIC_LOGS, insira 2 para ativar a alocação do arquivo de log dinâmico.
  4. Para CDR_MAX_DYNAMIC_LOGS, insira o número de pedidos de arquivo de log dinâmico que o ER pode fazer em uma sessão. Ou insira -1 para incluir logs indefinidamente.

Monitorando a Fila de Envio para um Nó

Monitore estatísticas sobre a fila de envio no nó atual em um domínio do Enterprise Replication (ER) e sobre os nós de destino e as réplicas para as quais o nó atual é uma origem de transações replicadas.

O ER coleta informações dos logs lógicos e coloca os dados a serem transferidos na fila de envio. Para monitorar a fila de envio, no menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Fila de Envio.

A página Fila de Envio exibe estatísticas sobre as transações da fila de envio. Ela fornece informações sobre o espaço de armazenamento, as transações enviadas ao nó de destino e a memória necessária para as transações na fila de envio. Fornece também informações sobre os nós de destino e as replicações das quais o nó atual é a origem de transações replicadas.

  • Para determinar quantas transações estão na fila e se elas estão sendo armazenadas, revise os campos Transações na Fila, Transações na Memória e Transações em Spool.
  • Para determinar se as transações estão sendo aplicadas ao nó de destino, revise o campo Posição do Log de Captura. Compare-o com o campo Dados Confirmados Recentes, que indica a posição do log lógico da última transação aplicada ao nó de destino.
  • Para determinar se algum nó de destino não está aplicando transações, revise o campo Nós Suspensos. Clique também na guia Nó de Destino para revisar o estado da conexão de cada nó de destino relacionado.
  • Para visualizar o estado dos nós de destino nos quais o nó atual replica os dados, clique na guia Nós de Destino.
  • Para visualizar as replicações do nó atual, clique na guia Replicacações.

Visualizando Nós de Destino e Réplicas Relacionadas ao Nó Atual

Visualize os nós de destino e as réplicas para as quais o nó atual em um domínio do Enterprise Replication (ER) é uma origem de transações replicadas.

No menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Fila de Envio e, em seguida, clique na guia Nós de Destino ou na guia Réplicas.

Monitorando a E/S de Rede para um Nó do Enterprise Replication

Monitore um resumo da E/S da rede para o nó atual em um domínio do Enterprise Replication (ER). Revise informações detalhadas sobre o nó atual e seus nós relacionados.

Para verificar a atividade de rede para o nó atual, no menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Resumo. Na página Resumo, a seção Rede exibe as seguintes informações:

  • O número de nós conectados
  • O número de mensagens enviadas pelo nó atual
  • O número de mensagens recebidas pelo nó atual
  • A taxa de rendimento no nó atual
  • O número de mensagens que não foram enviadas devido a problemas de conexão ou outros problemas

Para verificar o status da rede do nó atual e dos nós relacionados, no menu do OAT, clique em Replicação > Detalhes do Nó > Rede. A página Rede exibe informações sobre a atividade da rede com nós peer que possuem mensagens enviadas para o nó atual ou recebidas dele. Nomes de nós peer são listados como nomes de grupos, como definido no arquivo sqlhosts.

  • Para determinar a atividade da rede recente no nó atual, revise o gráfico Detalhes da Rede, que mostra o número de mensagens por período. Clique no ícone de relógio para visualizar ou alterar a taxa de atualização, que varia de 0 a 60 segundos.
  • Para determinar o estado da conexão de um nó peer, revise a coluna Estado.
  • Para determinar o estado da interface de rede (NIF) de um nó peer, revise a coluna Conexão.

Monitorando o Uso do Espaço em Disco de um Nó

Monitore como o espaço em disco está sendo usado no nó atual em um domínio do Enterprise Replication (ER).

No menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Uso de Disco. A página Uso do Disco exibe informações sobre as três áreas de espaço usadas pelo ER:

  • Sbspace de Dados da Linha: Os sbspaces que armazenam os dados a serem replicados. Eles são definidos no parâmetro CDR_QDATA_SBSPACE.
  • Dbspace do Registro de Transação: Os dbspaces que armazenam os registros de transação. Eles são definidos no parâmetro CDR_QHDR_DBSPACE, se estiverem definidos. Caso contrário, por padrão, os registros de transação são armazenados no dbspace raiz.
  • Sbspace de Paginação do Agrupador: Os sbspaces usados para manipulação de grandes transações. Eles são definidos primeiro pelo parâmetro SBSPACETEMP, se estiverem definidos; em seguida, o parâmetro SBSPACENAME, se estiver definido; caso contrário, pelo parâmetro CDR_QDATA_SBSPACE.

Importante:
Se esses espaços forem preenchidos, o ER é interrompido até que o problema que esteja causando o spool seja resolvido ou até que espaço adicional em disco seja alocado.

Para alterar os valores desses parâmetros, no menu do OAT, clique em Replicação > Detalhes do Nó > Configuração e clique em Editar Configuração.

Monitorando a Fila de Recebimento para um Nó

Monitore as transações enfileiradas e pendentes da fila de recebimento no nó atual em um domínio do Enterprise Replication (ER).

No menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Resumo. Revise a seção Fila de Recebimento, que lista o número de transações enfileiradas e pendentes.

  • As transações enfileiradas foram recebidas no nó atual de um nó peer e armazenadas na fila de recebimento.
  • Transações pendentes estão esperando para serem aplicadas no nó de destino.

Monitorando Como os Dados Replicados São Aplicados ao Nó de Destino

Monitore como os dados replicados são aplicados ao nó atual em um domínio do Enterprise Replication (ER).

Depois que um nó de destino recebe transações de outro nó e as armazena na fila de recebimento, o ER aplica as transações às tabelas no nó de destino.

Para visualizar um resumo de alto nível de como os dados replicados são aplicados ao nó atual, no menu do IBM OpenAdmin Tool (OAT) for Informix, clique em Replicação > Detalhes do Nó > Resumo. Revise a seção Aplicar.

  • Estatísticas sobre cada nó de origem e sobre como os dados replicados de cada nó são aplicados ao nó atual

  • No menu do OAT, clique em Replicação > Detalhes do Nó > ATS e RIS. Essas páginas listam os arquivos ATS e RIS usados para ressincronizar transações depois que houver falhas.

Definindo um Conjunto de Réplicas

Crie um novo conjunto de réplicas.

Um conjunto de réplicas combina várias réplicas para formar um conjunto que pode ser administrado como uma unidade. Você pode usar um único comando para iniciar, parar, suspender ou retomar todas as réplicas no conjunto.

Para definir um conjunto de réplicas:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Conjuntos de Réplicas.
  3. No menu Ações, clique em Definir Conjunto de Réplicas.
  4. Digite um nome para o conjunto de réplicas. O nome deve ser exclusivo e não pode ser igual ao nome de uma réplica.
  5. Selecione as réplicas para o conjunto.

    • Para remover réplicas, selecione-as na lista Este Conjunto de Réplicas e clique em Remover ou em Remover Todas.

  7. Para configurar a opção de frequência para o conjunto de réplicas, clique em Avançar, ou para criar o conjunto de réplicas com o valor padrão, clique em Concluir.
  8. Opcional: Especifique a Frequência de replicação. O padrão é imediatamente quando os dados chegam.
  9. Para criar o conjunto de réplicas, clique em Concluir.

Definindo uma Réplica

Defina os participantes da réplica e seus atributos, incluindo a frequência de replicação de dados e como lidar com conflitos.

Com a ação Definir Réplica, você define uma réplica principal estrita e seus participantes. Uma réplica principal verifica se tabelas replicadas em servidores diferentes possuem tipos de dados de colunas consistentes. Uma réplica principal estrita também verifica se as tabelas replicadas possuem nomes de colunas idênticos. Todos os servidores de banco de dados que contêm réplicas principais devem ser capaz de estabelecer uma conexão direta com o servidor de banco de dados da réplica principal.

Pré-requisito:
Um tabela deve ter uma chave primária para ser participante da replicação. Além disso, ela deve ser uma tabela real. Ela não pode ser um sinônimo, uma visualização ou uma sequência. Além disso, não pode ser uma tabela externa. Somente as tabelas que podem ser participantes da replicação aparecem na lista de tabelas selecionáveis.

Para definir uma réplica e os participantes da réplica:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Réplicas.
  3. No menu Ações, clique em Definir Réplica.
  4. Digite um nome exclusivo para a réplica.
  5. Selecione o grupo de servidores a ser usado como a base da replicação principal a partir da lista Servidor de Replicação Principal.
  6. Para criar tabelas automaticamente no servidor de destino existentes no servidor de réplicas principais, mas não no servidor de destino, selecione Criar Tabelas Ausentes nos Servidores de Destino.
  7. Usando os filtros, selecione as tabelas e os servidores de replicação que você deseja definir como participantes da réplica.
    1. No Filtro de Banco de Dados, digite todo ou parte do nome do banco de dados.
    2. No Filtro de Tabela, digite todo ou parte do nome da tabela. Continue filtrando a lista até que uma único participante seja exibido para cada servidor.
    3. Para selecionar todos os participantes especificados pelos filtros, clique em Selecionar.

      Você também pode selecionar ou limpar uma tabela clicando nela na lista.

    Nota:
    O servidor de réplicas principais deve ser um participante da nova réplica. Quando você seleciona as tabelas para participarem da réplica, deve selecionar uma tabela no servidor de réplicas principais.
  8. Especifique o tipo de participante: primário ou receber apenas. O padrão é primário: o participante envia e recebe dados. Para alterar a opção, clique no ícone.
  9. Especifique quais privilégios serão usados para aplicar os dados replicados. O padrão é proprietário da tabela. Para alterar a opção, clique no ícone.
  10. Para configurar as opções avançadas para a réplica, clique em Avançar, ou para criar a réplica com os valores padrão para as opções avançadas, clique em Concluir.

    O ER verifica se as tabelas em diferentes servidores possuem tipos de dados de colunas consistentes e nomes de colunas idênticos. Se Avançar e Concluir não estiverem disponíveis, as tabelas não estarão consistentes.

  11. Opcional: Especifique as opções avançadas:
    • Escopo da Resolução de Conflitos. O padrão é transação.
    • Regra de Resolução de Conflitos. O padrão é registro de data e hora.
    • Formato de Ponto Flutuante. O padrão é IEEE.
    • Ativar Aborted Transaction Spooling (ATS). Por padrão, o ATS não está ativado.
    • Ativar Row Information Spooling (RIS). Por padrão, o RIS não está ativado.
    • Replicar Linha Completa. Por padrão, linhas completas são replicadas. Esta opção está ativada.
    • Disparar Acionadores no Destino. Por padrão, acionadores não são disparados na tabela de destino.
    • Reter Linhas Excluídas. Por padrão, as linhas excluídas não são retidas.
    • Frequência de replicação. O padrão é imediatamente quando os dados chegam.
  12. Para criar a réplica, clique em Concluir.

Usando Modelos para Configurar o Enterprise Replication

Use modelos para configurar e implementar o Enterprise Replication.

Um modelo usa informações de esquemas sobre um banco de dados, um grupo de tabelas, colunas e chaves primárias para definir um grupo de réplicas principais e um conjunto de réplicas. Primeiro, defina uma modelo e depois o instancie nos servidores onde você deseja replicar dados.

Definindo um Modelo

Crie um modelo. Especifique as tabelas que serão usadas para o modelo, o banco de dados e o servidor Enterprise Replication (ER) no qual as tabelas estão localizadas e as opções de modelo.

Um modelo define um grupo de réplicas principais e um conjunto de réplicas. Um modelo consiste de informações de esquemas sobre um banco de dados, um grupo de tabelas, atributos de colunas e chaves primárias que identificam linhas.

Para definir um modelo:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Modelos.
  3. No menu Ações, clique em Definir Novo Modelo.
  4. Digite um nome para o modelo. O nome deve ser exclusivo e não pode ser igual ao nome da réplica ou do conjunto de réplicas.
  5. Na lista Servidores, expanda os nomes de servidor e bancos de dados para exibir as tabelas.
    Dica:
    Para filtrar a lista, comece digitando o nome de servidor na caixa de texto Filtro de Servidor e o nome de banco de dados na caixa de texto Filtro de Banco de Dados.
  6. Selecione as tabelas para o modelo.
    • Para incluir todas as tabelas de um banco de dados na lista, selecione o banco de dados na lista Tabelas e clique em Incluir.
    • Para incluir uma tabela específica, selecione a tabela e clique em Incluir.
    • Para remover uma tabela do modelo, selecione-a e clique em Remover ou em Remover Tudo.
  7. Para criar o modelo com as opções padrão, clique em Concluir ou em Avançar, para especificar uma das opções a seguir, e depois clique em Concluir.
    • Escopo da Resolução de Conflitos. O padrão é transação.
    • Regra de Resolução de Conflitos. O padrão é ignorar.
    • Formato de Ponto Flutuante. O padrão é IEEE.
    • Ativar Aborted Transaction Spooling (ATS). Por padrão, o ATS não está ativado.
    • Ativar Row Information Spooling (RIS). Por padrão, o RIS não está ativado.
    • Disparar Acionadores no Destino. Por padrão, acionadores não são disparados na tabela de destino.
    • Reter Linhas Excluídas. Por padrão, as linhas excluídas não são retidas.

Agora você pode instanciar o modelo nos servidores de banco de dados do domínio ER usando a ação Concretizar Modelo.

Importante:
Para replicar dados em um domínio ER com base nas réplicas do modelo, você deve instanciar o modelo em pelo menos dois servidores.

Concretizando um Modelo

Instancie um modelo nos servidores Enterprise Replication (ER).

Pré-requisito:
Antes de poder instanciar um modelo, você deve defini-lo. Consulte Definindo um Modelo

Para instanciar um modelo:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Modelos.
  3. Na lista de modelos, selecione o modelo que será instanciado.
  4. No menu Ações, clique em Concretizar Modelo.
  5. Na lista Servidores Disponíveis, selecione os servidores nos quais instanciar o modelo.
    Dica:
    Comece digitando o nome do servidor na caixa de texto Filtro de Servidor para filtrar a lista.
  6. Para instanciar o modelo nos servidores com as opções padrão, clique em Concluir ou em Avançar, para especificar uma das opções a seguir, e depois clique em Concluir.
    • Criar Tabelas Ausentes. Quando esta opção é selecionada, tabelas existentes no servidor de réplicas principais, mas não no servidor de destino, são criadas automaticamente nos servidores de destino. Esta opção é selecionada por padrão. O dbspace das tabelas criadas é configurado por padrão para o mesmo dbspace do banco de dados.
    • Aplicar Dados Replicados com Privilégios do Proprietário da Tabela. Especifique se o proprietário da tabela de origem é o proprietário das tabelas criadas quando o modelo é concretizado. Por padrão, as tabelas pertencem ao usuário informix.
    • Configurar o Servidor para Apenas Receber. Especifique se os servidores de destino serão servidores que apenas receberão. Por padrão, os servidores enviam e recebem dados replicados.
    • Sincronizar Dados. Especifique um servidor para atuar como a origem da sincronização de dados nos servidores nos quais o modelo está instanciado. Especifique como identificar linhas nos servidores de destino que não estão presentes no servidor de origem de dados. Por padrão, linhas extras são excluídas dos servidores de destino.

Excluindo um Modelo

Exclua um modelo do Enterprise Replication (ER) e qualquer conjunto de réplicas criado quando o modelo foi instanciado.

Importante:

Para excluir um modelo:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Modelos.
  3. Selecione o modelo.
  4. No menu Ações, clique em Excluir Modelo.

Iniciando um Conjunto de Réplicas

Altere o estado de todas as réplicas em um conjunto de réplicas para ativo.

Quando uma réplica está ativa, o Enterprise Replication (ER) captura dados do log lógico no servidor de origem e os transmite para os participantes ativos nos servidores de destino.

Para iniciar um conjunto de réplicas:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Conjuntos de Réplicas.
  3. Clique no conjunto de réplicas na tabela. Para visualizar uma lista de réplicas no conjunto, expanda o nome do conjunto de réplicas.
  4. No menu Ações, clique em Iniciar Conjunto de Réplicas.
  5. Na página Iniciar Conjunto de Réplicas, selecione os participantes que irão iniciar.
    • Para iniciar todos os participantes no conjunto de réplicas, selecione a caixa de opção Todos os Participantes.
    • Para iniciar participantes específicos, selecione a caixa de opção para cada participante que você deseja iniciar. Os participantes são especificados pelo servidor de banco de dados.
  6. Para executar uma sincronização inicial das réplicas que você está iniciando, execute estas etapas:
    1. Selecione a caixa de opção Sincronizar Dados.
    2. Selecione o servidor de origem de dados para a sincronização a partir da lista de servidores.
    3. Especifique como identificar linhas no servidor de destino que não estão presentes no servidor de origem de dados.
      • Excluir: Exclua as linhas extras do servidor de destino.
      • Manter: Mantenha as linhas extras no servidor de destino.
      • Mesclar: Mantenha as linhas extras nos servidores de destino e as replique para o servidor de origem.
  7. Para iniciar o conjunto de réplicas, clique em OK.

    • Para parar participantes específicos, selecione a caixa de opção para cada participante que você deseja parar. Os participantes são especificados pelo servidor de banco de dados.
  6. Para parar o conjunto de réplicas, clique em OK.

Suspendendo um Conjunto de Réplicas

Suspenda todas as réplicas em um conjunto de réplicas.

Quando uma réplica está em um estado suspenso, o Enterprise Replication captura e acumula alterações no banco de dados de origem, mas não transmite os dados capturados para o banco de dados de destino.

Importante:
Quando um conjunto de réplicas é suspenso, o Enterprise Replication mantém os dados replicados na fila de envio até que o conjunto seja retomado. Se uma grande quantidade de dados for gerada para as réplicas no conjunto enquanto ele estiver suspenso, o espaço da fila de envio poderá ser preenchido, causando perda de dados. O Enterprise Replication não sincronizará transações se uma réplica em um conjunto de réplicas estiver suspensa. Por exemplo, uma transação que atualiza as tabelas X e Y será dividida se a replicação da tabela X for suspensa.

Para suspender um conjunto de réplicas:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Conjuntos de Réplicas.
  3. Clique no conjunto de réplicas na tabela. Para visualizar uma lista de réplicas no conjunto, expanda o nome do conjunto de réplicas.
  4. No menu Ações, clique em Suspender Conjunto de Réplicas. Na página Suspender Conjunto de Réplicas, a lista exibe os participantes no conjunto de réplicas, especificados pelo servidor de banco de dados. Todos os participantes são selecionados, pois a suspensão de uma réplica faz com que todos os participantes entrem no estado suspenso.
  5. Para suspender o conjunto de réplicas, clique em OK.

Sincronizando Novamente Dados Entre Servidores de Replicação

Corrija inconsistências de dados entre tabelas replicadas. Verifique inconsistências de dados, repare inconsistências durante uma verificação, sincronize dados ou repare transações com falha com arquivos Aborted Transaction Spooling (ATS) e Row Information Spooling (RIS).

Com o IBM Informix Replication Plug-in for OpenAdmin Tool (OAT), é possível reparar inconsistências de dados entre tabelas replicadas das formas a seguir:

  • Verificar a consistência e reparar linhas inconsistentes: Uma verificação de consistência compara os dados entre um servidor de referência e um ou mais servidores de destino e gera um relatório que descreve todas as inconsistências. Você pode optar por reparar linhas inconsistentes durante uma verificação de consistência. Você pode executar uma verificação de consistência e uma sincronização otimizada em uma única réplica ou em um conjunto de réplicas.
  • Sincronizar dados: A sincronização replica cada linha na réplica especificada ou conjunto de réplicas a partir do servidor de referência para todos os servidores de destino especificados. Você pode usar a sincronização direta para preencher um novo servidor de destino ou um servidor de destino existente que tenha se tornado muito inconsistente.
  • Reparar transações usando arquivos ATS ou RIS: Uma reparação usando um arquivo ATS ou RIS repara as linhas associadas à única transação gravada no arquivo ATS ou RIS especificado.
Importante:
Executar uma verificação de consistência ou sincronização pode consumir uma grande quantidade de espaço em seus arquivos de log. Verifique se você possui espaço suficiente antes de executar este comando.

Reparando Transações com Falha com Arquivos ATS

Repare transações com falha usando arquivos Aborted Transaction Spooling (ATS). A tarefa de reparação do ATS reconcilia linhas que falharam quando aplicadas ao servidor de destino.

Pré-requisito:
Crie um diretório para armazenar os arquivos ATS, ative a geração de arquivos ATS no servidor Enterprise Replication (ER) e ative a geração de arquivos ATS para a réplica.

Um arquivo ATS é gerado quando uma transação replicada falha quando aplicada ao banco de dados de destino devido a um conflito e toda a transação falha. Quando você repara a transação usando o arquivo ATS, repara as linhas associadas à transação que foram registradas no arquivo ATS.

Para reparar transações usando arquivos ATS:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia ATS.
  3. Na tabela Arquivos Aborted Transaction Spooling (ATS), marque as caixas de seleção para os arquivos ATS.

    Se você não reparar todos os arquivos ATS disponíveis ao mesmo tempo, algumas tarefas de reparação poderão falhar devido a problemas ao manter os relacionamentos entre as tabelas (limitadores de integridade referencial).

  4. Clique em Reparar.
  5. Reveja os arquivos ATS selecionados para reparação.

    Para ver os detalhes de um arquivo ATS, clique no nome dele.

  6. Clique em Reparar.
  7. Monitore o progresso da tarefa e reveja os resultados na lista Status da Tarefa de Reparação do ATS.

    Para exibir os detalhes de uma tarefa, clique no nome de arquivo na lista.

Depois de executar uma verificação de consistência, uma verificação e uma reparação ou uma tarefa de sincronização em uma réplica, o status da tarefa e os resultados são exibidos e atualizados na página Status da Tarefa.

Para monitorar as tarefas:

  1. No menu do IBM OpenAdmin Tool (OAT) for Informix, expanda Replicação e, em seguida, clique em Réplicas.
  2. Clique na guia Status da Tarefa.
  3. Selecione Verificação de Réplica ou Status de Sincronização. A lista exibe o status das tarefas. Você pode clicar em um título da coluna para classificar a lista.
  4. Para exibir mais detalhes sobre uma tarefa, clique no nome dela.

    Na página Detalhes da Tarefa de Réplica, clique no botão Atualizar para atualizar os detalhes de uma tarefa que está em execução.

Criando um Acordo de Nível de Serviço

Crie um acordo de nível de serviço para uma grade, um cluster ou um conjunto de réplicas usando um Gerenciador de Conexões.

Pré-requisitos:
Para o Gerenciador de Conexões, o IBM OpenAdmin Tool (OAT) for Informix requer o Informix 11.70.xC3 e o Gerenciador de Conexões que está empacotado com o Informix 11.70.xC3 ou com o IBM Informix Client Software Development Kit (Client SDK) 3.70.xC3. A grade, o cluster ou o conjunto de réplicas deve ser definido como uma unidade de conexão para um Gerenciador de Conexões.

Um Acordo de Nível de Serviço (SLA) é um contrato entre aplicativos clientes especificados e seus provedores de serviços, neste caso, servidores de banco de dados Informix. Com base nos acordos de nível de serviço, o Gerenciador de Conexões conecta os aplicativos clientes aos servidores apropriados. O Gerenciador de Conexões aceita pedidos de conexão do cliente e os redireciona para o servidor adequado ou age como um servidor proxy entre os aplicativos clientes e os servidores.

Um Gerenciador de Conexões fornece balanceamento de carga e direciona as solicitações de conexão do cliente ao servidor apropriado em uma unidade de conexão baseada nos Acordos de Nível de Serviço (SLA). Quando a qualidade de dados é monitorada, o Gerenciador de Conexões pode rotear conexões do cliente para os servidores de replicação com base no número de falhas de aplicação, na latência de transação e na carga de trabalho. Quando a qualidade de dados não é monitorada, o Gerenciador de Conexões pode rotear conexões do cliente com base apenas na carga de trabalho. Para especificar as falhas de aplicação e a latência em um SLA, ative o monitoramento da qualidade de dados modificando um servidor que seja um membro da grade na página Replicação > Domínio do ER.

Para criar um acordo de nível de serviço:

  1. No menu do OAT, clique em Replicação > Gerenciador de Conexões.
  2. Clique no Gerenciador de Conexões.
  3. Clique na unidade de conexão na lista.
  4. No menu Ações, clique em Criar um SLA.
  5. Conclua o assistente.
